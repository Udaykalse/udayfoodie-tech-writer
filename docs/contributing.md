# Contributing Guide

## Welcome

Thank you for your interest in contributing to the **UdayFoodie Landing Page** project.

Whether you're fixing a bug, improving the documentation, or suggesting a new feature, your contributions are appreciated.

---

# Ways to Contribute

You can contribute by:

- Fixing bugs
- Improving documentation
- Optimizing HTML
- Improving CSS
- Enhancing responsive design
- Improving accessibility
- Optimizing images
- Reporting issues
- Suggesting new features

---

# Prerequisites

Before contributing, ensure you have:

- Git installed
- A GitHub account
- Visual Studio Code (recommended)
- Basic knowledge of HTML and CSS

---

# Getting Started

## 1. Fork the Repository

Create your own copy of the repository on GitHub.

---

## 2. Clone Your Fork

```bash
git clone https://github.com/YOUR_USERNAME/udayfoodie-landing-page.git
```

---

## 3. Navigate to the Project

```bash
cd udayfoodie-landing-page
```

---

## 4. Create a New Branch

```bash
git checkout -b feature/your-feature-name
```

Example:

```bash
git checkout -b feature/improve-navbar
```

---

## 5. Make Your Changes

Possible improvements include:

- HTML updates
- CSS improvements
- Better documentation
- Responsive fixes
- Image optimization

---

## 6. Test Your Changes

Before committing:

- Verify the website loads correctly.
- Check that all images display.
- Ensure the layout remains responsive.
- Confirm there are no broken links.

---

## 7. Commit Your Changes

```bash
git add .
git commit -m "feat: improve navigation layout"
```

Use clear and meaningful commit messages.

Examples:

```text
docs: update installation guide
fix: correct image path
style: improve button spacing
feat: add responsive navigation
```

---

## 8. Push Your Branch

```bash
git push origin feature/your-feature-name
```

---

## 9. Create a Pull Request

Open a Pull Request describing:

- What changed
- Why the change was made
- Screenshots (if applicable)

---

# Coding Guidelines

When contributing:

- Write semantic HTML.
- Keep CSS organized.
- Use meaningful class names.
- Avoid duplicate code.
- Keep documentation up to date.

---

# Documentation Guidelines

Documentation should be:

- Clear
- Concise
- Grammatically correct
- Beginner-friendly
- Written in Markdown

---

# Reporting Issues

When opening an issue, include:

- Problem description
- Steps to reproduce
- Expected behavior
- Actual behavior
- Browser information
- Screenshots (if available)

---

# Code of Conduct

Please be respectful and constructive when collaborating.

Treat all contributors professionally and communicate respectfully.

---

# Thank You

Thank you for helping improve this project.

Every contribution—large or small—is valuable.
